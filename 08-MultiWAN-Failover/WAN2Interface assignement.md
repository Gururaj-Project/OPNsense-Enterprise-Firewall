
# Assigning the Second WAN Interface

Before configuring Multi-WAN, assign the additional Ethernet interface to OPNsense.

Navigate to:

Interfaces
    └── Assignments

Add the new interface and rename it as **WAN2**.

Enable the interface and configure:

| Option | Example |
|----------|----------|
| Interface | WAN2 |
| IPv4 Configuration | Static / DHCP (As provided by ISP) |
| IPv4 Address | ISP Assigned |
| Gateway | ISP2 Gateway |

Save and Apply the configuration.

Verify that the WAN2 interface receives connectivity before proceeding.

---

# Configuring Gateways

Navigate to:

System
    └── Gateways
            └── Configuration

Create gateways for both ISP connections.

Example:

| Gateway | Purpose |
|----------|----------|
| ISP1_GW | Primary Internet Connection |
| ISP2_GW | Secondary Internet Connection |

Configure a Monitor IP for each gateway.

Example:

| Gateway | Monitor IP |
|----------|-------------|
| ISP1_GW | 8.8.8.8 |
| ISP2_GW | 8.8.4.4 |

The Monitor IP is periodically checked by OPNsense to determine gateway health.

---

# Configuring Gateway Groups

Navigate to:

System
    └── Gateways
            └── Groups

Create a new Gateway Group.

Example:

| Gateway | Priority |
|----------|----------|
| ISP1_GW | Tier 1 |
| ISP2_GW | Tier 2 |

Trigger Level:

Member Down

During implementation, **Member Down** was selected to verify automatic failover. Depending on network requirements, the trigger level can later be changed to Packet Loss or High Latency.

Save and Apply.

---

# Configuring Automatic Failover

After creating the Gateway Group, OPNsense automatically monitors both gateways.

If the Tier 1 gateway becomes unavailable, traffic is redirected to Tier 2.

When Tier 1 becomes available again, traffic automatically returns to the preferred gateway.

No manual intervention is required during normal failover operation.

# Updating Firewall Rules

After configuring the Gateway Group, update the firewall rules that provide internet access for the internal VLANs.

Navigate to:

Firewall
    └── Rules
            ├── VLAN10
            └── VLAN20

Edit the internet access rule.

Change:

Gateway

Default

To:

Gateway

Gateway Group

Repeat the same configuration for every VLAN that requires internet connectivity.

# Configuring DNS Firewall Rules

After updating the internet access firewall rules to use the Gateway Group, DNS traffic must also be allowed on each internal VLAN interface.

Navigate to:

```
Firewall
    └── Rules
            ├── VLAN10
            └── VLAN20
```

Create a firewall rule similar to the following.

| Field | Example |
|---------|----------|
| Action | Pass |
| Protocol | TCP/UDP |
| Source | VLAN Network |
| Destination | This Firewall |
| Destination Port | Domain (53) |
| Description | Traffic Routing to DNS |

Apply the same configuration to every VLAN that requires DNS services.

This rule allows client devices to send DNS requests to the OPNsense firewall for domain name resolution.

Without this rule, users may be able to reach public IP addresses directly, but domain names may fail to resolve correctly.

> **Note**
>
> The exact interface and destination may vary depending on your DNS design. The objective is to ensure that DNS traffic generated by OPNsense itself is routed correctly and is not unintentionally affected by policy-based routing.

---

# Policy-Based Routing

After updating the internet access firewall rules, all outbound traffic from VLAN10 and VLAN20 is routed through the configured Gateway Group instead of the default gateway.

This enables OPNsense to automatically apply the configured failover policy.

Only firewall rules responsible for internet-bound traffic should use the Gateway Group.

Inter-VLAN communication should continue using normal routing unless a different network design requires policy-based routing.

---

# Important Configuration Notes

During my implementation, changing the firewall rule gateway directly to the Gateway Group without verifying supporting firewall rules resulted in temporary loss of GUI access and internet connectivity.

To avoid this situation:

- Keep console access available.
- Verify Anti-Lockout rules.
- Configure DNS routing rules for firewall-generated traffic where required.
- Apply changes carefully and test each stage before proceeding.

---

> **Important**

Only modify the firewall rules responsible for internet access by changing the Gateway from **Default** to the configured **Gateway Group**.

Rules used for inter-VLAN communication should continue using normal routing unless your network design specifically requires policy-based routing between VLANs.
