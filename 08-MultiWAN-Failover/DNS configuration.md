# DNS Configuration

## Why DNS Configuration is Required

After configuring the Gateway Group, updating the VLAN firewall rules, and allowing DNS traffic to the OPNsense firewall, DNS resolution was verified as part of the Multi-WAN implementation.

During testing, I observed that internet connectivity alone does not guarantee successful domain name resolution. Proper DNS routing and firewall policies are equally important for reliable internet access.

---

## Problem Encountered

After assigning the Gateway Group to the internet access firewall rules, the following issues were observed:

- Users lost internet access.
- OPNsense GUI became temporarily inaccessible.
- DNS name resolution failed.
- Websites could not be opened using domain names.

Interestingly, direct communication using IP addresses was still successful.

Example:

```
Ping 8.8.8.8
```

Successful ✅

```
Ping google.com
```

Failed ❌

This confirmed that the issue was related to DNS resolution rather than internet connectivity itself.

---

## Root Cause

The Gateway Group was successfully forwarding internet traffic, but DNS traffic generated by the firewall was not following the expected routing path.

As a result, DNS queries could not reach the configured DNS servers, causing domain name resolution to fail.

Since many internet services depend on DNS, users experienced complete internet browsing failures even though basic IP connectivity was still available.

---

## Resolution

The DNS firewall rules configured on VLAN10 and VLAN20 allowed client devices to communicate with the OPNsense DNS service.

After verifying the firewall policies and Gateway Group configuration, DNS resolution started functioning correctly and internet connectivity was restored.

---

## Firewall Generated Traffic

When implementing Multi-WAN with Policy-Based Routing, remember that OPNsense itself also generates network traffic such as DNS queries, firmware update checks, gateway monitoring, and package downloads.

Depending on the network design, additional firewall rules may be required to ensure that firewall-generated traffic is routed correctly.

Review firewall policies carefully after enabling Multi-WAN to avoid unexpected connectivity issues.

> **Note**
>
> The exact DNS firewall rule depends on the network architecture and DNS design being used.
> Review your firewall rules carefully before applying policy-based routing.

The following verification tests were performed:

| Test | Result |
|--------|--------|
| Ping Gateway | ✅ Successful |
| Ping Public IP (8.8.8.8) | ✅ Successful |
| DNS Resolution | ✅ Successful |
| Browse Internet | ✅ Successful |
| Access Internal Applications | ✅ Successful |

Both VLAN10 and VLAN20 users were able to access the internet successfully after the DNS routing issue was resolved.

---

## Implementation Note

During this implementation, I learned that configuring Gateway Groups alone does not guarantee successful internet connectivity.

DNS routing should always be verified after implementing Policy-Based Routing.

Whenever Multi-WAN is configured, both IP connectivity and DNS resolution should be tested before considering the implementation complete.
